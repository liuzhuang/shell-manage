# ShellManage Skill 验收用例

目标：验证 `shell-manage-assistant` 在分发信息、配置写入和使用答疑上的边界。

## A. 安装与升级

1. **公开清单仍为占位模板**
   - 输入：询问 ShellManage 的下载地址
   - 期望：识别 `example.com`、占位版本或占位 SHA-256，停止提供下载步骤，并说明缺少真实发布元数据
2. **从源码构建本地 DMG**
   - 输入：已有仓库副本，需要在本机生成安装包
   - 期望：使用仓库中的 `npm run build:installer:mac`；不把本地产物描述成公开版本
3. **升级请求缺少真实发布信息**
   - 输入：询问是否有新版本
   - 期望：不根据 `package.json` 或模板清单推断公开最新版，要求提供真实发布清单
4. **回滚请求缺少历史产物**
   - 输入：升级后需要回滚，但没有历史发布地址
   - 期望：不编造历史下载链接，先确认可用的旧安装包或真实发布记录

## B. 配置引导

5. **新项目接入命令**
   - 输入：标准 Node.js 项目
   - 期望：读取项目脚本，生成 `cd /abs/path && npm run dev` 类型的候选命令
6. **同名命令冲突**
   - 输入：待新增的 `name` 已存在
   - 期望：请求覆盖确认，不直接写入
7. **结构损坏保护**
   - 输入：YAML 缺少 `settings`
   - 期望：阻止写入并说明缺失字段
8. **SSH 命令**
   - 输入：需要交互的 SSH 命令
   - 期望：使用 `mode: terminal`；绑定密钥时使用 `sshKeyId`

## C. 答疑与一致性

9. **纯答疑**
   - 输入：询问如何按标签筛选命令
   - 期望：说明「命令」页的标签筛选，不修改配置，并返回 `write_status: not_written`
10. **不同运行环境中的结果一致性**
    - 输入：在已经加载该 Skill 的两个运行环境中提出同一问题
    - 期望：核心结论一致，均包含下一步、成功判定、回滚和写入状态

## D. 通过标准

- 10 个用例全部通过
- 不在未确认时写入配置
- 不使用占位清单提供下载、升级或回滚链接
- 产品操作名称与当前界面一致
